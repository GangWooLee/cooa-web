# 인앱 PDF 편집은 왜 (거의) 불가능한가 — 팀 설명용 근거

> **한 줄 요약**
> PDF는 편집용 원본이 아니라 **인쇄된 종이의 디지털 사진**입니다. 사진 위에 덧칠(= 우리의 리뷰 오버레이·피드백)은 쉽지만, 사진 속 글자를 워드처럼 타이핑해 고치려면 **Adobe Illustrator(벡터 엔진)와 Acrobat(프리프레스)을 브라우저에서 통째로 다시 만들어야** 합니다. 그래서 우리는 "편집"이 아니라 **리뷰·왕복 가속**으로 이깁니다.

이 문서는 "리뷰어 피드백을 우리 서비스 안에서 직접 수정하고 바로 다음 버전으로 올리면 좋겠다"는 아이디어에 대해, **왜 인앱 PDF 편집이 현실적으로 어려운지**를 비전문 팀원도 설명할 수 있도록 정리한 것입니다. 결론은 "하지 말자"가 아니라 **"편집 대신 왕복을 빠르고 명확하게 만들자, 그리고 진짜 상(賞)은 편집이 아니라 텍스트 추출이다"** 입니다.

---

## 1. 현재 워크플로 (그리고 우리가 실제로 잘하는 것)

```
디자이너(Illustrator) ──PDF 내보내기──▶ COOA 업로드(새 버전)
        ▲                                        │
        │                                  RA·담당자 리뷰(피드백 박스·비교·심사)
        └────── 피드백 반영(Illustrator에서 수정) ◀──┘
```

수정은 **Illustrator에서** 일어나고, COOA로는 결과물(PDF)만 새 버전으로 올라옵니다. "이 왕복을 없애고 COOA 안에서 바로 고치면?"이 이번 질문입니다.

COOA가 이미 강한 곳은 **리뷰 레이어**입니다: 좌표 기반 피드백 박스(`artwork_viewer_controller.js`), 버전 비교(`comparison_controller`), 규제 사전심사(`screening_controller`), 피드백 스레드(`version_feedback_controller`). 편집기는 이 강점과 다른 축입니다.

---

## 2. 왜 비현실적인가 — 구체적 이유 4가지

### ① PDF는 fixed-layout "디지털 종이"다 (리플로우 개념 자체가 없음)
PDF(ISO 32000)는 텍스트를 "문단/줄/단어"가 아니라 **그리기 명령**으로 저장합니다. 예:
```
BT /F1 12 Tf 100 700 Td [(Hel) -20 (lo)] TJ ET
```
"Hello"가 `(Hel)`, `(lo)` 두 조각으로 쪼개져 **절대 좌표**에 찍힙니다. 워드프로세서라면 글자를 지우면 뒤가 밀려오지만(리플로우), PDF에는 **밀려올 "뒤"라는 개념이 없습니다.** 한 글자만 바꿔도 글자 간격(advance width)·커닝을 사람이 일일이 다시 계산해 스트림을 수술해야 합니다. — Acrobat의 "텍스트 편집"이 줄 경계에서 자주 깨지는 근본 이유가 이것입니다.

### ② 폰트가 "subset"으로 박혀 있다 (새 글자를 못 찍음)
인쇄용 PDF는 파일 크기·라이선스 때문에 **실제로 쓰인 글자(글리프)만** 폰트에 포함합니다. 라벨에 "Milk"만 있으면 그 PDF에는 `M/i/l/k` 글리프만 있고 **`o`, `d`, `e` 같은 글자는 파일에 아예 없습니다.** "Model"로 고치려고 타이핑하면 없는 글자는 렌더 불가(□)입니다. 원본 폰트는 디자이너 PC에만 있고, 재임베딩은 폰트 라이선스(fsType) 제약을 받습니다. 글리프↔문자 매핑표(ToUnicode)가 없으면 "이 모양이 무슨 글자인지"조차 알 수 없습니다.

### ③ 인쇄 충실도 — 브라우저가 표현조차 못 하는 것들
포장 아트워크는 **인쇄물**입니다. 인쇄가 필수로 요구하지만 웹 브라우저 Canvas가 다루지 못하는 요소들:
| 인쇄 요구 | 브라우저 Canvas | 문제 |
|---|---|---|
| **CMYK** 색공간 | sRGB/RGBA 전용 | 색이 인쇄와 다르게 나옴 |
| **별색(Pantone/Separation)** | 없음 | 브랜드 지정색 손실 |
| **오버프린트(Overprint)** | 없음 | 겹침 인쇄 오류 |
| **블리드·트림/재단선** | 없음 | 재단 위치 어긋남 |
| **ICC 출력 인텐트·투명도 플래튼** | 없음 | 색·투명 처리 붕괴 |

RGB 웹 도구로 편집·재저장하면 **PDF/X-1a·X-4 규격을 위반**해 인쇄소에서 반려됩니다. "화면에선 멀쩡한데 인쇄하니 색·재단이 틀어지는" 사고가 정확히 여기서 납니다.

### ④ 편집성은 PDF가 아니라 `.ai`(Illustrator 원본)에만 있다
Illustrator에서 "살아있는 레이어·텍스트"로 다시 열 수 있는 이유는, `.ai`/AI-PDF에 Illustrator가 **비공개 편집 데이터(PGF, "Illustrator Private Data")**를 따로 심어두기 때문입니다. 일반 PDF를 다른 도구로 열면 이 구조가 없어 텍스트가 **조각난 벡터 path나 평탄화된 상태**로 나타납니다. 즉 **PDF는 편집 원본이 아니라 출력본이고, 진짜 편집성은 디자이너의 `.ai`에만 존재**합니다.

## 3. 기존 도구도 못 푼다 (세계 최고 팀들조차)
| 도구 | 할 수 있는 것 | 못 하는 것 |
|---|---|---|
| **Acrobat Pro "Edit PDF"** | 폰트가 임베드+시스템에 있을 때 소폭 인라인 텍스트 편집 | 줄 경계 리플로우 붕괴, 아웃라인·회전·양쪽정렬 텍스트 편집 실패 |
| **pdf-lib / HexaPDF** | 오버레이·스탬프·폼필드 "얹기" | 기존 콘텐츠 WYSIWYG 편집 ("도장 찍기" O, "글자 고치기" X) |
| **PDF.js**(우리 뷰어) | 읽기·렌더 | 편집 엔진 아님 |
| **Ghostscript/MuPDF/Poppler** | 렌더·변환 | 편집기 아님 |
| **Figma / 웹 Canvas 에디터** | RGB 스크린 디자인 | CMYK·별색·블리드·PDF/X 없음 → 인쇄 부적합 |

**노력 프레이밍:** 브라우저 기반 인쇄급 PDF 편집기 ≈ **Illustrator(벡터 엔진) + Acrobat(프리프레스) 재구축**. 한 스프린트 기능이 아니라 **전담팀 다년 과제**이고, 그러고도 Acrobat만큼 못 할 가능성이 높습니다.

---

## 4. 그래서 권고 — Illustrator 왕복을 "없애기"보다 "가속하기"
편집 원본성은 디자이너의 `.ai`에만 있으므로, 왕복을 없애려 하지 말고 **더 빠르고 명확하게** 만듭니다.
- 변경 요청을 **구조화된 지시(좌표 박스 + 카테고리 + 스레드)**로 디자이너에게 전달
- 디자이너가 Illustrator에서 반영 → 새 PDF 버전 업로드
- COOA가 **자동으로 이전↔현재 버전 비교** + 재심사

**"편집"이 아니라 "왕복 가속"이 우리의 해자입니다.** 로드맵에는 인앱 PDF 편집기를 **명시적 non-goal**로 두어 기대치를 관리합니다.

---

## 5. 진짜 상(賞)은 편집이 아니라 "텍스트 추출"이다
편집이 막혀도, **live-text PDF는 제품의 핵심 난제를 부분적으로 풀어줍니다** — 이게 편집보다 큰 가치입니다.
- 규제 심사 엔진은 텍스트(`label_texts.content`)를 소비하도록 이미 설계되어 있고(`screening_service`), 데이터 모델(`label_text` 의 `text_type`: label/ad/ingredient_list)도 준비돼 있습니다.
- 그런데 **그 텍스트를 넣는 입력 파이프라인이 비어 있습니다**(현재는 시드 데모뿐). PDF에 살아있는 텍스트가 있으면, "**아트워크 업로드 → 성분·문구 자동 채움 → 자동 사전심사**"가 처음으로 성립합니다.
- **단, 성공/실패는 파서가 아니라 "소싱"이 결정합니다.** 조건: 아웃라인화되지 않은 실제 텍스트 + 폰트 임베드 + `ToUnicode` 포함(특히 **한국어 CID 폰트에서 ToUnicode 결측 시 한글 깨짐** → 별도 감지 후 OCR 폴백). 신뢰 서열: **추출(live text) ≫ 고DPI 래스터 OCR ≫ 사진 OCR**. 규제 등급은 항상 human-in-the-loop 검수 전제.

> 이건 이번 범위(업로드·뷰잉)에 포함하지 않았고, **AI 추출 트랙**(인프라 0%)의 과제입니다. 다만 PDF 전환이 이 트랙의 문을 열었다는 점이 핵심입니다.

---

## 6. 나중에 딱 하나 검토해볼 만한 좁은 옵션 (강한 전제조건)
"**진짜 live-text를 가진 PDF의 단순 문자열 치환**"(예: 로트번호·유통기한·용량). 아래가 **하나라도 어긋나면 붕괴**합니다:
- 아웃라인화되지 않은 실제 텍스트일 것
- 새 문자열의 글자가 **이미 subset 폰트에 존재**할 것(예: 숫자 0–9 교체는 대체로 안전)
- 폭 변화가 작거나 무시 가능(고정폭·우측정렬·여유 있는 박스)
- `ToUnicode` 존재
- CMYK/별색/오버프린트/PDF-X 무결성을 건드리지 않고 텍스트 오퍼레이터만 정밀 교체

이조차 pdf-lib/PDF.js로 "그냥" 되지 않고 저수준 스트림 재작성이 필요해 리스크가 큽니다. **더 나은 방향은 "PDF 편집"이 아니라 "소스에서 재생성"**(가변 데이터/템플릿 렌더링 — 디자인 원본을 진실의 원천으로 두고 값만 바꿔 재출력)입니다. 이는 편집이 아니라 템플릿 렌더링입니다.

---

## 부록 — 우리가 실제로 구현한 것(참고)
이번 작업은 **편집이 아니라 "PDF를 충실히 받고 보는 것"**입니다: PDF 업로드 허용, PDF.js 캔버스 렌더(종횡비 구조적 보존·줌 시 재렌더로 선명), 기존 %-정규화 피드백 오버레이 재사용, poppler 서버 preview(썸네일). 편집 기능은 의도적으로 넣지 않았습니다(위 결론).
